## 1. Введение

### 1.1 Цель документа
Настоящий документ содержит детальное описание архитектуры, конфигурационных параметров и системных интерфейсов управляющего подпакета SmartApi. Документ определяет правила инициализации фреймворка, маршрутизации запросов, политик безопасности, логирования и многобазовой оркестрации, являясь руководством для DevOps-инженеров и системных архитекторов платформы умного дома.

### 1.2 Область применения
Объектом описания является исключительно папка SmartApi, содержащая входные точки приложения (wsgi.py, asgi.py), глобальный маршрутизатор (urls.py) и декларативный конфигурационный манифест (settings.py). Данный компонент связывает изолированное приложение бизнес-логики api с операционной средой, базами данных, веб-серверами и внешним IAM-провайдером.
### 1.3 Определения, акронимы и сокращения

| Термин | Определение                                                                                                               |
|------|--------------------------------------------------------------------------------------------------------------------------|
| WSGI  | Web Server Gateway Interface (стандарт синхронного взаимодействия Django с веб-сервером). |
| ASGI  | Asynchronous Server Gateway Interface (асинхронный стандарт для поддержки WebSockets и Long-Polling).        |
| CORS / Allowed Hosts  | Политики ограничения сетевого доступа к ресурсу по доменным именам.        |
| Middleware  | Промежуточное программное обеспечение (обработчики сквозных процессов).        |
| Multi-DB Routing  | Динамическая маршрутизация SQL-запросов между изолированными СУБД.        |

### 1.5 Обзор документа
Документ декомпозирует настройки ядра SmartApi, описывает жизненный цикл прохождения входящего HTTP/HTTPS запроса через каскад Middleware-классов, а также специфицирует роутинг баз данных и профилирование логирования системы.

## 2. Обзор архитектурного решения

### 2.1 Интересы заинтересованных сторон
- DevOps-инженеры и Системные Администраторы: Требуют гибкой контейнеризации, поддержки переменных окружения (dotenv), предсказуемого логирования в stdout/файлы и выделенного эндпоинта проверки работоспособности (Health Check).

- Специалисты по Информационной Безопасности: Требуют жесткой изоляции доверенных доменов (ALLOWED_HOSTS), валидации JWT-токенов Keycloak на системном уровне и прозрачного логирования HTTP-активности.

- Разработчики API: Требуют автоматической генерации OpenAPI/Swagger документации на основе зарегистрированных приложений.

### 2.2 Выбранные ракурсы архитектуры
#### 2.2.1 Точка зрения контекста
Smart API функционирует как центральный шлюз оркестрации. К внешним границам системы относятся:
* **Входной слой:** Мобильное приложение жильца и Web-панель управления УК (через Nginx).
* **Слой инфраструктуры связи:** IP-АТС Asterisk (интеграция по DBAPI к MariaDB).
* **Слой медиапотоков:** Видеокамеры и домофонные панели, подключенные по RTSP через прокси-сервер MediaMTX.
* **Слой оборудования:** Аппаратные контроллеры реле дверей, принимающие команды по протоколу MQTT.
![Use Case](diagrams/2.2.1.drawio.png)


#### 2.2.2 Точка зрения состава
Система декомпозирована по принципу чистой архитектуры на технические модули (директории) внутри папки `api/`:
* `controller`: Содержит классы DRF ViewSets и эндпоинты, обрабатывающие входящие HTTP-запросы и сериализующие ответы.
* `service`: Ядро бизнес-логики системы, где инкапсулированы правила проверки доступов, генерация кодов и оркестрация вызовов.
* `repository`: Слой доступа к данным, изолирующий логику прямых SQL-запросов и сложных выборок от бизнес-логики.
* `models`: Декларативное описание таблиц и связей базы данных средствами Django ORM.
* `infrastructure`: Шлюзы, драйверы и низкоуровневые клиенты для интеграции со сторонними сервисами (Keycloak, Asterisk PJSIP, MQTT, Push-серверы).
* `middleware`: Компоненты перехвата запросов (логирование, проверка JWT-токенов, обработка контекста пользователей).
* `management`: Кастомные административные команды управления (включая скрипты автоматической инициализации таблиц).
* `unit_tests`: Изолированная среда автоматизированного тестирования компонентов и слоев приложения.
![Component](diagrams/2.2.2.drawio.png)

#### 2.2.3 Точка зрения логической структуры
Взаимодействие компонентов внутри системы следует строгому направлению зависимостей:
* **Слой представлений (`controller`):** Вызывает методы исключительно из слоя `service`.
* **Слой бизнес-логики (`service`):** Оперирует чистыми правилами, запрашивая данные через абстракции слоя `repository` и используя интерфейсы `infrastructure`.
* **Слой репозиториев (`repository`):** Напрямую работает с классами из `models` или выполняет сырые SQL-вызовы к внешним БД через драйверы `infrastructure`.
![Diagram](diagrams/2.2.3.drawio.png)

#### 2.2.4 Точка зрения структуры связей и физической топологии
Описывает физическое размещение элементов системы, аппаратную конфигурацию, топологию локальной сети жилого комплекса и физические ограничения среды СКУД:

* **Серверный узел (Локальный сервер УК / Облако):** Выделенный сервер архитектуры x86-64 под управлением Ubuntu Server, на котором в изолированном Docker-контуре функционируют контейнеры Smart API, PostgreSQL, Redis и прокси-сервер MediaMTX.
* **Физическая сеть ЖК:** Закрытая виртуальная локальная сеть (VLAN), изолированная от публичного интернета, что гарантирует защиту от несанкционированного доступа к оборудованию.
* **Аппаратный слой СКУД:** Физические контроллеры (Device), управляющие реле замков дверей и калиток, подключенные к локальной сети по Ethernet/Wi-Fi и взаимодействующие с сервером по протоколу MQTT через выделенный брокер.
* **Аппаратный слой мультимедиа:** Домофонные панели (вызов по SIP/PJSIP через АТС Asterisk) и уличные/подъездные IP-видеокамеры (трансляция RTSP-потоков высокого разрешения).
![Diagram](diagrams/2.2.4.drawio.png)

#### 2.2.5 Точка зрения зависимостей
Жизненный цикл Smart API зависит от внешних систем, однако архитектура изолирует их отказы:
* **Жесткая зависимость:** СУБД PostgreSQL и Кэш Redis. При их недоступности API возвращает `HTTP 503`.
* **Слабая зависимость (Изолированная):** АТС Asterisk и Push-провайдеры. При падении АТС компоненты `infrastructure` изолируют и логируют ошибку связи, но функции генерации кодов в `service` продолжают работу.
![Diagram](diagrams/2.2.5.drawio.png)

#### 2.2.6 Модель базы данных
Слой `models` декларативно описывает структуру хранения данных через классы Django ORM, полностью отражающие схему физической БД и исключающие избыточность:
* **`Building`**: Модель для учета физических строений и жилых корпусов.
* **`Entrance`**: Подъезды и точки входа, связанные с конкретным зданием через поле `building`.
* **`Facility`**: Конечные объекты недвижимости (квартиры, офисы), привязанные к конкретному подъезду через поле `entrance`.
* **`FacilityUser`**: Связующая таблица для реализации связи "многие-ко-многим", сопоставляющая объект `facility` и уникальный идентификатор пользователя `user_id` (хранится в виде UUID, транслируемого из Keycloak) с разграничением ролей (собственник/арендатор).
* **`Device`**: Физическое оборудование (контроллеры СКУД, считыватели, реле замков), привязанное к точке прохода через поле `entrance`.
* **`Camera`**: Реестр IP-видеокамер, хранящий RTSP-ссылки на видеопотоки и метаданные; связывается с объектами через `building` или `entrance`.
* **`AccessCode`**: Сущность кодов доступа (PIN-коды, RFID-метки). Валидация обеспечивается полями `code` (строковый ключ), `is_active` (булев флаг) и временными рамками `start_time` и `end_time`.
* **`Call`**: Журнал вызовов с домофонных панелей, регистрирующий входящие события, `device_id`, `user_id` и статус соединения.
* **`PassLog`**: Журнал мониторинга событий СКУД. Фиксирует факты прохода через поля `device_id`, `user_id` (или ссылку на использованный `AccessCode`) и точную временную метку создания записи `created_at`.
![Diagram](diagrams/2.2.6.drawio.png)

#### 2.2.7 Точка зрения интерфейса
Взаимодействие с внешними клиентами стандартизировано:
* **REST API:** Конечные точки для мобильных приложений, объявленные в слое `controller`. Данные принимаются и отдаются исключительно в формате `application/json`.
* **Контрактная документация:** Спецификация интерфейса автоматически формируется пакетом `drf-spectacular` по стандарту OpenAPI 3.0 и доступна на эндпоинте `/swagger/`.
![Diagram](diagrams/2.2.7.drawio.png)

#### 2.2.8 Точка зрения взаимодействия
Сценарий «Открыть дверь из мобильного приложения»:
1. Клиент отправляет HTTP POST запрос в слой `controller`.
2. `middleware` перехватывает запрос, валидирует подпись JWT и извлекает UUID пользователя.
3. Контроллер передает управление в слой `service`.
4. Сервис запрашивает права через репозиторий, проверяя связь в таблице `Facilities_Users`. При успешной проверке отправляется команда в MQTT-клиент слоя `infrastructure`.
5. `RequestLoggingMiddleware` фиксирует структурированную запись об успешном открытии в `smart_api.log`.
![Diagram](diagrams/2.2.8.drawio.png)

#### 2.2.9 Точка зрения алгоритмов
Ключевые вычислительные алгоритмы системы:
* **Алгоритм проверки прав доступа:** Реализован в слое `service`. Выполняет рекурсивный обход дерева объектов недвижимости, проверяя, привязана ли квартира пользователя к подъезду или зданию, за которым закреплено вызываемое устройство.
![Diagram](diagrams/2.2.9.1.drawio.png)
* **Алгоритм безопасной синхронизации схем данных:** Реализован в команде `ensure_api_tables` (слой `management`). Алгоритм считывает метаданные текущей БД, сопоставляет со структурами классов из `models` и выполняет безопасные SQL-команды `ALTER TABLE` только для неблокирующих полей, предотвращая `table lock`.

#### 2.2.10 Точка зрения динамики состояния
Жизненный цикл и валидность доступа в сущности `AccessCode` определяются в слое `service` на основе логического сочетания полей модели в базе данных:
* **Проверка активности:** Выполняется методом валидации кода, который сопоставляет текущую системную метку времени с диапазоном `start_time` <= `now` <= `expired_at` при обязательном условии установленного флага `is_active == True`.
* **Прекращение действия доступа:** Происходит либо автоматически (когда текущее время превышает `expired_at`), либо принудительно путем явного обновления записи в БД инструкцией `is_active = False` через методы слоя `service` при аннулировании кода оператором.
![Diagram](diagrams/2.2.10.drawio.png)

#### 2.2.11 Точка зрения параллелизма
Параллельное выполнение и масштабирование системы организовано на двух уровнях:
* **Синхронный HTTP-слой:** Обработка входящих веб-запросов масштабируется за счет запуска нескольких независимых воркеров Gunicorn (WSGI-процессы) для слоя `controller`.
* **Асинхронный IO-слой:** Массовая отправка push-сигналов вызова вынесена из синхронного контекста Django в асинхронный event-loop библиотеки `asyncio` (`aioapns`) внутри слоя `infrastructure`, что предотвращает блокировку пула HTTP-потоков.
![Diagram](diagrams/2.2.11.drawio.png)

#### 2.2.12 Точка зрения развертывания
Система развертывается в изолированном контейнеризированном окружении через Docker Compose:
* Контейнер `smart_api_web`: Запуск приложения под управлением Gunicorn.
* Контейнер `smart_api_nginx`: Балансировщик трафика, раздающий статику и проксирующий запросы на воркеры.
* Контейнеры инфраструктуры: Изолированные в приватной подсети `smart_network` образы `postgres:15-alpine` и `redis:7-alpine`. Разделение конфигураций тестирования и продакшна реализовано файлами `docker-compose.test.yml` и `docker-compose.prod.yml`.
![Diagram](diagrams/2.2.12.drawio.png)

#### 2.2.13 Шаблоны проектирования и архитектурные стили
Описывает повторно используемые архитектурные шаблоны, стили и паттерны проектирования, которые определяют структуру компонентов системы и правила их взаимодействия.

* **Многоуровневая архитектура**
  * **Где применен:** Структура исходного кода бэкенда Smart API.
  * **Суть:** Проект разделен на изолированные слои. Зависимости направлены строго сверху вниз. Нижние слои ничего не знают о верхних.

* **Издатель-Подписчик**
  * **Где применен:** Интеграция с физическим оборудованием СКУД через `mosquitto_mqtt`.
  * **Суть:** Бэкенд Smart API не держит постоянное прямое соединение с сотнями контроллеров на дверях. Вместо этого он публикует (Publish) команды открытия в топики брокера, а контроллеры, подписанные (Subscribe) на свои уникальные топики, мгновенно их считывают. Это обеспечивает слабую связанность и высокую масштабируемость.

* **Асинхронный обработчик задач**
  * **Где применен:** Фоновые процессы в связке Django + Celery.
  * **Суть:** Тяжелые или непредсказуемые по времени операции (отправка мобильных Push-уведомлений, обращение к внешнему SIP-серверу Asterisk) изолируются в виде отдельных задач и передаются воркерам, гарантируя стабильность основного REST API.
![Diagram](diagrams/2.2.13.drawio.png)
  
## 3. Представления архитектурного дизайна

### 3.1 Структурное представление (Structural View)
Ядро SmartApi представляет собой декларативный оркестратор, собирающий воедино системные библиотеки и доменное приложение умного дома.

### 3.1.1 Компонентная структура файлов ядра
- settings.py (Декларативный манифест): Считывает переменные среды через .env, инициализирует структуру директорий (автоматическое создание папки logs/ на уровне BASE_DIR), настраивает подключение драйвера django.db.backends.mysql и внедряет сторонние экосистемные плагины (django_restql, drf_yasg, api).

- urls.py (Глобальный диспетчер маршрутов): Реализует разделение периметров трафика. Регистрирует системный изолированный эндпоинт /health/ для балансировщика нагрузки (Nginx/Traefik) и делегирует весь остальной бизнес-трафик во внутреннее приложение по контракту path('api/v1/', include('api.urls')).

### 3.1.2 Структура многобазовой конфигурации
В ядре объявлена дефолтная база данных MySQL, динамические реквизиты которой (DB_NAME, DB_USER и т.д.) извлекаются из переменных операционной системы через os.environ.get. Настройка строгого режима "init_command": "SET sql_mode='STRICT_TRANS_TABLES'" гарантирует защиту от невалидных типов данных при транзакциях.

Связующим звеном архитектуры данных выступает инструкция: DATABASE_ROUTERS = ['api.router.AdminRouter']
Она активирует выделенный класс-маршрутизатор, который перехватывает SQL-операции ядра и прозрачно перенаправляет запросы телефонии или административной логики в соответствующие целевые СУБД (включая закомментированный шаблон admin_db).

## 3.3 Интерфейсное представление (Interface View)
Ядро SmartApi предоставляет внешнему инфраструктурному периметру (Docker-контейнерам и оркестраторам) три основных интерфейса взаимодействия:

1. Синхронный шлюз веб-сервера (wsgi.py):
   - Реализация: Переменная application = get_wsgi_application().
   - Потребитель: Производственный веб-сервер Gunicorn (внутри Docker-окружения). Служит для обработки стандартных REST-запросов со средним временем жизни потока.
2. Асинхронный событийный шлюз (asgi.py):
   - Реализация: Переменная application = get_asgi_application().
   - Потребитель: Асинхронный сервер Daphne / Uvicorn. Используется в умном доме для обслуживания долгоживущих WebSocket-соединений.
3. Интерфейс проверки живучести (Health Endpoint):
   - Контракт: GET /health/ $\longrightarrow$ JsonResponse({"status": "ok"}).
   - Потребитель: Механизмы Health Check оркестратора Docker-Compose / Kubernetes и апстримы прокси-сервера Nginx. Не требует авторизации и обращения к базе данных, подтверждая физическую доступность самого Python-процесса ядра.

## 4. Архитектурные решения

#### ID: 001-Keycloak-Edge-Auth
- Название: Вынос аутентификации на системный уровень ядра REST Framework.
- Контекст: Недопустимо реализовывать проверку валидности сессий жильцов локально в каждом отдельном контроллере views.py.
- Решение: Внедрение класса api.Infrastructure.integrations.keycloak.keycloak_auth.KeycloakTokenAuth на уровне ядра гарантирует, что ни один запрос не дойдет до бизнес-логики, если его заголовок Authorization: Bearer не пройдет криптографическую проверку подлинности в Keycloak.

#### ID: 002-Secure-SSL-Proxy-Header
- Название: Доверие заголовкам обратного проксирования (Reverse Proxy Trust).
- Контекст: Так как приложение SMART-API разворачивается внутри Docker, внешний SSL-трафик терминируется на Nginx (папка gateway), а внутрь контейнера Django запросы приходят по обычному HTTP. Без спец-настройки Django будет генерировать некорректные абсолютные ссылки (http вместо https).
- Решение: Переменная SECURE_PROXY_SSL_HEADER = ('HTTP_X_FORWARDED_PROTO', 'https') указывает ядру считать запрос безопасным (HTTPS), если прокси-сервер Nginx передал соответствующий заголовок X-Forwarded-Proto.
